# Bug Brief: landing-page — Active-member CTA routes to /home instead of /membership

Date: 2026-04-05 17:18

## Failing Test
Spec: `frontend/e2e/landing-page.spec.ts`
Test name: `LAND-04 signed-in user with membership sees /membership as the primary action`

## Failure

```
Error: expect(locator).toHaveAttribute(expected) failed

Locator:  getByRole('link', { name: 'Open member area' })
Expected: "/membership"
Received: "/home"
Timeout:  5000ms

Call log:
  - waiting for getByRole('link', { name: 'Open member area' })
    9 × locator resolved to <a href="/home" ...>Open member area</a>
      - unexpected value "/home"

  at frontend/e2e/landing-page.spec.ts:186:72
```

## Steps to Reproduce

1. Navigate to `http://localhost:3000/` with an active GymFlow auth session seeded
   in localStorage (`gymflow-auth`) and the `/api/v1/memberships/me` endpoint
   returning a 200 with an ACTIVE membership.
2. Observe the hero primary CTA: a link labelled "Open member area".
3. Inspect the `href` attribute of that link.

**Observed:** `href="/home"`
**Expected:** `href="/membership"` (per PRD AC-5 and AC-5 of the landing-page PRD)

The mock for `**/api/v1/memberships/me` returned HTTP 200 with an ACTIVE

destination is the sole defect.

## Severity

**Critical** — the core auth-aware routing for active members is broken. A signed-in
member with an active membership is sent to `/home` (an unspecified or wrong route)
instead of `/membership` (the member area). PRD AC-5 is not satisfied.
