# Review: onboarding-unified-signup — 2026-04-23

## Blockers (must fix before PR)

- [x] `frontend/src/pages/onboarding/OnboardingPage.tsx:29-34` — After the terms-step `register` succeeds, `setTokens` + `setUser` flip `isAuthenticated` true, which (a) makes `useBootstrap` in `App.tsx` set `bootstrapLoading = true` and (b) makes `OnboardingRoute` swap `OnboardingShell` for `BootstrapSpinner`. When bootstrap resolves, `OnboardingPage` remounts and its mount effect runs `computeResumeStep` again. For the now-authenticated user with `firstName/lastName/phone/dob` populated in the store, `computeResumeStep` returns `'membership'`, overwriting the `currentStep = 'done'` that `OnboardingShell.handleContinue` had just set. `OnboardingRoute` then sees `onboardingCompletedAt !== null && currentStep !== 'done'` and redirects to `/home`. The user lands authenticated but never sees the Done screen — **AC-07 (the guest is taken through the existing `done` step)** is broken end-to-end. **Fix applied in commit `2842888`**: `computeResumeStep` now early-returns `'done'` when `store.currentStep === 'done'`, so the bootstrap-spinner remount cannot override the wizard's intent. SDD §4.4 updated to document the rule. E2E spec `e2e/specs/onboarding-unified-signup.spec.ts` re-run: 1 passed.

## Suggestions (non-blocking)

- `frontend/src/components/onboarding/MiniNav.tsx:52` — Eyebrow renders `STEP 01 · ACCOUNT · 1 of 6` as a single inline string. The handoff prototype (`design_reference/index.html` line 635-646) shows two stacked lines: the eyebrow `STEP 01 · ACCOUNT` in `--color-primary-light`, with the fraction `1 of 7` underneath in `--color-fg-muted`. The current treatment also uses `--color-fg-metadata` for the whole line, losing the eyebrow's primary-light accent. Restructure to two lines per handoff.
- `frontend/src/components/onboarding/steps/StepCredentials.tsx:154-164` — The "Required" pill uses `--color-primary-light` text + `--color-primary-tint` background + `rounded-full`, matching sibling steps (StepProfile) but **not** the handoff (handoff §1 / prototype `.pill-required`: `--color-fg-muted` text, `1px --color-border-card` border, `--radius-md` rectangular pill, no fill). Sibling consistency is reasonable but the handoff is explicit. Either update the pill or update the handoff.
- `frontend/src/components/onboarding/steps/StepCredentials.tsx:33-42` — The focus `useEffect` has an empty deps array (and an `eslint-disable-next-line`). On a snap-back where the credentials step is *already* the active step (e.g. user navigated back manually before the late error fires — edge case), the late-error change won't re-trigger the focus + scroll. Adding `credentialsLateError` to deps (and gating the unconditional focus path) makes the snap-back behaviour deterministic.
- `frontend/src/components/onboarding/steps/StepCredentials.tsx:201,246` — Email and password labels include a `*` asterisk in `--color-error-fg`. The handoff prototype (`design_reference/index.html` line 768, 794) uses plain labels without the asterisk; the REQUIRED pill on the headline already conveys the gating. The colored asterisk also reads as a permanent error tint on a healthy field. Drop the asterisks.
- `frontend/src/components/onboarding/steps/StepDone.tsx:16-35` — The bootstrap-spinner detour (caused by the blocker above) also unmounts and remounts StepDone. `completeFiredRef` is a per-mount ref, so `POST /onboarding/complete` can fire twice. Backend idempotency is unconfirmed. Once the blocker is fixed the remount goes away, but a belt-and-braces guard keyed on `localStorage` or on `onboardingCompletedAt !== null` (which it already short-circuits on) would harden this.
- `frontend/src/components/onboarding/OnboardingShell.tsx:353` — `continueRetry={!!termsError}` flips the footer button to "Try again →" whenever any terms error is set, including transient ones. The SDD §4.5 only describes a retry affordance for `INVALID_*` codes (snap-back handles `EMAIL_ALREADY_EXISTS` separately). Consider scoping `continueRetry` to the codes that actually produce a recoverable retry on the same step.
- `frontend/src/store/onboardingStore.ts:117-118` — `setCredentials` always overwrites `password` even when called with the same string the user typed across mounts (e.g. step navigation re-mounting StepCredentials on back-nav). This needlessly bumps `lastTouchedAt`, extending the 24h plaintext-password window. A no-op short-circuit when both args match the existing state would tighten the guarantee.
- `frontend/src/components/onboarding/OnboardingShell.tsx:33-38` — `toE164` is duplicated inline; the same helper exists in `StepProfile.tsx:20`. Extract to `frontend/src/utils/phone.ts` (or similar) to keep one source of truth for the format.
- `frontend/src/store/onboardingStore.ts:62` — `defaultState.currentStep = 'credentials'` is correct for guests but means an authenticated user whose store was hard-reset (clearAuth → reset) lands on a step they cannot meaningfully complete (already authenticated, no register call to fire). `OnboardingPage`'s `computeResumeStep` patches this at runtime, but defaulting `currentStep` to `'profile'` for authenticated mounts and `'credentials'` for unauthenticated would remove the patch.

## Iteration 2 — Manual self-audit (Lesson 13) found 2 additional blockers

Reproduced live against the dev stack with Playwright MCP.

- [x] **`frontend/src/components/onboarding/steps/StepProfile.tsx:44-105` — Continue silently fails on the profile step.** Both desktop (`hidden lg:grid`) and mobile (`lg:hidden`) `<StepContent>` are mounted simultaneously, sharing the same `profileRef`. Each onChange in StepProfile calls `store.setProfileFields(...)`, which re-renders `OnboardingShell` (the parent) → both StepProfile instances re-render → the mobile instance's `useImperativeHandle` runs LAST in JSX order → `profileRef.current.submit` is the mobile instance's closure, reading mobile's empty local `useState` (because `display:none` removes the mobile inputs from the a11y tree, so input events go only to the visible desktop instance). Validation fails on empty firstName/lastName/phone/dob → returns false → no advance, no error visible. Identical race to the documented terms-case fix at `OnboardingShell.tsx:162-167`. **Fix applied in commit `<this commit>`**: `StepProfile.submit()` now reads field values from `useOnboardingStore.getState()` (the store always has the latest values from the onChange sync) instead of from local `useState`. SDD §6 Decision 12 documents the rule for future step authors. Verified live: profile step now advances without uploading a photo.

- [x] **`frontend/src/hooks/useHomePage.ts:91-95` — Infinite loop on `GET /api/v1/memberships/me` for users with no active membership.** The mount effect's deps `[activeMembership, membershipLoading, fetchMyMembership]` re-trigger after each fetch because the store's `set({ membershipLoading: false, ... })` flips `membershipLoading` from `true → false`. The guard `!activeMembership && !membershipLoading` is true again → fetch again → 50,000+ calls/min observed in the dev browser. The unified-signup flow makes this a normal terminal state because completing onboarding without selecting a plan is now a supported path. **Fix applied in commit `<this commit>`**: include `membershipErrorCode` in the destructure and the guard — short-circuit the effect once the store records `NO_ACTIVE_MEMBERSHIP`. SDD §6 Decision 13 documents this. Verified live: console errors dropped from 198 to 2 (one expected 404, one unrelated 403 on `/trainers/favorites`).
