# Review: onboarding-unified-signup ‚Äî 2026-04-23

## Blockers (must fix before PR)

- [ ] `frontend/src/pages/onboarding/OnboardingPage.tsx:29-34` ‚Äî After the terms-step `register` succeeds, `setTokens` + `setUser` flip `isAuthenticated` true, which (a) makes `useBootstrap` in `App.tsx` set `bootstrapLoading = true` and (b) makes `OnboardingRoute` swap `OnboardingShell` for `BootstrapSpinner`. When bootstrap resolves, `OnboardingPage` remounts and its mount effect runs `computeResumeStep` again. For the now-authenticated user with `firstName/lastName/phone/dob` populated in the store, `computeResumeStep` returns `'membership'`, overwriting the `currentStep = 'done'` that `OnboardingShell.handleContinue` had just set. `OnboardingRoute` then sees `onboardingCompletedAt !== null && currentStep !== 'done'` and redirects to `/home`. The user lands authenticated but never sees the Done screen ‚Äî **AC-07 (the guest is taken through the existing `done` step)** is broken end-to-end. Fix: in `computeResumeStep`, treat `currentStep === 'done'` as authoritative (early-return `'done'`), or guard the resume override behind `currentStep !== 'done'`. Either way, `setStep` must not run when the store already says `done`.

## Suggestions (non-blocking)

- `frontend/src/components/onboarding/MiniNav.tsx:52` ‚Äî Eyebrow renders `STEP 01 ¬∑ ACCOUNT ¬∑ 1 of 6` as a single inline string. The handoff prototype (`design_reference/index.html` line 635-646) shows two stacked lines: the eyebrow `STEP 01 ¬∑ ACCOUNT` in `--color-primary-light`, with the fraction `1 of 7` underneath in `--color-fg-muted`. The current treatment also uses `--color-fg-metadata` for the whole line, losing the eyebrow's primary-light accent. Restructure to two lines per handoff.
- `frontend/src/components/onboarding/steps/StepCredentials.tsx:154-164` ‚Äî The "Required" pill uses `--color-primary-light` text + `--color-primary-tint` background + `rounded-full`, matching sibling steps (StepProfile) but **not** the handoff (handoff ¬ß1 / prototype `.pill-required`: `--color-fg-muted` text, `1px --color-border-card` border, `--radius-md` rectangular pill, no fill). Sibling consistency is reasonable but the handoff is explicit. Either update the pill or update the handoff.
- `frontend/src/components/onboarding/steps/StepCredentials.tsx:33-42` ‚Äî The focus `useEffect` has an empty deps array (and an `eslint-disable-next-line`). On a snap-back where the credentials step is *already* the active step (e.g. user navigated back manually before the late error fires ‚Äî edge case), the late-error change won't re-trigger the focus + scroll. Adding `credentialsLateError` to deps (and gating the unconditional focus path) makes the snap-back behaviour deterministic.
- `frontend/src/components/onboarding/steps/StepCredentials.tsx:201,246` ‚Äî Email and password labels include a `*` asterisk in `--color-error-fg`. The handoff prototype (`design_reference/index.html` line 768, 794) uses plain labels without the asterisk; the REQUIRED pill on the headline already conveys the gating. The colored asterisk also reads as a permanent error tint on a healthy field. Drop the asterisks.
- `frontend/src/components/onboarding/steps/StepDone.tsx:16-35` ‚Äî The bootstrap-spinner detour (caused by the blocker above) also unmounts and remounts StepDone. `completeFiredRef` is a per-mount ref, so `POST /onboarding/complete` can fire twice. Backend idempotency is unconfirmed. Once the blocker is fixed the remount goes away, but a belt-and-braces guard keyed on `localStorage` or on `onboardingCompletedAt !== null` (which it already short-circuits on) would harden this.
- `frontend/src/components/onboarding/OnboardingShell.tsx:353` ‚Äî `continueRetry={!!termsError}` flips the footer button to "Try again ‚Üí" whenever any terms error is set, including transient ones. The SDD ¬ß4.5 only describes a retry affordance for `INVALID_*` codes (snap-back handles `EMAIL_ALREADY_EXISTS` separately). Consider scoping `continueRetry` to the codes that actually produce a recoverable retry on the same step.
- `frontend/src/store/onboardingStore.ts:117-118` ‚Äî `setCredentials` always overwrites `password` even when called with the same string the user typed across mounts (e.g. step navigation re-mounting StepCredentials on back-nav). This needlessly bumps `lastTouchedAt`, extending the 24h plaintext-password window. A no-op short-circuit when both args match the existing state would tighten the guarantee.
- `frontend/src/components/onboarding/OnboardingShell.tsx:33-38` ‚Äî `toE164` is duplicated inline; the same helper exists in `StepProfile.tsx:20`. Extract to `frontend/src/utils/phone.ts` (or similar) to keep one source of truth for the format.
- `frontend/src/store/onboardingStore.ts:62` ‚Äî `defaultState.currentStep = 'credentials'` is correct for guests but means an authenticated user whose store was hard-reset (clearAuth ‚Üí reset) lands on a step they cannot meaningfully complete (already authenticated, no register call to fire). `OnboardingPage`'s `computeResumeStep` patches this at runtime, but defaulting `currentStep` to `'profile'` for authenticated mounts and `'credentials'` for unauthenticated would remove the patch.
