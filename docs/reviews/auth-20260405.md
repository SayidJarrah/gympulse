## Suggestions (non-blocking)

- **Dead error-code branches in `useAuth.ts` login handler** — Lines 91–94 handle `REFRESH_TOKEN_EXPIRED` and `REFRESH_TOKEN_INVALID` inside the `login` catch block. The `/auth/login` endpoint can never return these codes; they are only surfaced by the Axios interceptor. The dead branches set visible error messages instead of silently redirecting, which contradicts the design spec's Flow 3 behaviour. Remove them to reduce confusion and future maintenance cost.

- **`authStore` persists `isAuthenticated` to localStorage causing a stale-auth flash** — On page reload, `isAuthenticated` reads `true` from localStorage even when the access token has since expired. While the Axios interceptor recovers transparently, the UI briefly renders the authenticated shell before the 401 cycle completes. Consider initialising `isAuthenticated` to `false` on store hydration and deriving it from the access-token's presence+expiry instead, or performing a silent refresh on app mount. Documented in SDD Section 7 as a known trade-off.

- **Admin seed password committed as a comment in V3 migration** — `V3__seed_admin_user.sql` contains "the placeholder value 'Admin@1234'" in the SQL comment. `V5__fix_admin_seed_password.sql` removes the comment-based reference but V3 still has it. The SDD open task (Section 5, db-architect checklist) requires a rotation runbook before any non-local environment is provisioned. Document the bcrypt regeneration procedure (one shell command) in `docs/runbook.md` and redact the password from the V3 comment.

- **`docs/design/auth.md` missing Benchmark Citation** — The design-standards skill requires every screen design to include a `Benchmark:` citation. The auth design spec has no such section. Add a citation (e.g. "Benchmark: Vercel login — centered card on dark field, wordmark above card, minimal cognitive overhead") to complete the spec and satisfy the design-standards requirement.

- **`useAuth` reads `refreshToken` from top-level Zustand hook destructure** — `useAuth.ts:61` destructures `refreshToken` from `useAuthStore()`. Because `refreshToken` changes whenever the Axios interceptor rotates tokens, this causes `useAuth` subscribers to re-render on every token rotation even when they only care about `user` or `isLoading`. Use `useAuthStore.getState().refreshToken` inside the `logout` callback (same pattern already used in `LoginPage.tsx:13`) to avoid unnecessary re-renders.

- **Concurrent refresh queue: `isRefreshing` and `pendingRequests` are module-level mutable state** — `axiosInstance.ts:31-35`. Module-level mutable variables survive hot-module replacement in Vite dev mode, which can cause the pending-queue to hold stale closures across HMR cycles. In production this is harmless, but during development it occasionally manifests as requests that never resolve after a hot reload. Encapsulate the queue in a closure or class to make resets safe.
